### 3.1 ConfigService 路径逻辑

**文件**: `src/utils/config.ts`

```typescript
// getProjectLockPath 实现
public getProjectLockPath(): string {
  return join(this.getSkillsInstallDir(), ".skill-lock.json");
}

// getSkillsInstallDir 优先级
// 1. WOPAL_SKILLS_DIR 环境变量
// 2. settings.jsonc 中 space.skillsDir
// 3. 默认值: .wopal/skills (相对于 workspace)
```

### 3.2 LockManager 路径

**文件**: `src/utils/lock-manager.ts`

```typescript
constructor(configService: ConfigService) {
  this.projectLockPath = configService.getProjectLockPath();
  this.globalLockPath = path.join(os.homedir(), ".agents", ".skill-lock.json");

```

### 3.3 Logger 行为

**文件**: `src/utils/logger.ts`

```typescript
// logger.info() 只在 debug 模式输出到控制台
info(message: string, data?: any): void {
  if (this.isDebug) {
    console.log(logLine.trim());
  }
  // 始终写入日志文件

```

**重要结论**: 用户可见消息必须使用 `console.log()`，不能用 `logger.info()`

## 六、需遵守的规则

### 6.1 测试原则

1. **环境变量必须显式传递**: execSync 不会继承 process.env，必须通过 `env: { ...process.env, ... }` 传递
2. **Mock 路径必须与 CLI 一致**: 测试中 mockConfigService 的路径必须与 CLI 的 getConfig() 返回的路径一致
3. **用户消息用 console.log**: Logger.info 只在 debug 模式可见
4. **集成测试超时设置**: 
   - execSync timeout: 30s (涉及 GitHub API)
   - it() 测试超时: 60s

### 6.2 代码规范

- Prettier 格式化，2 空格缩进
- 导入路径使用 `.js` 扩展名 (ESM)
- 测试文件命名 `*.test.ts`

## 九、决策依据

1. **为什么先修复现有测试**: 确保基线稳定，避免后续工作被阻塞
2. **为什么用 console.log 替代 logger.info**: Logger.info 只在 debug 模式输出到控制台，集成测试无法捕获
3. **为什么 execSync 需要 env 参数**: 子进程不继承父进程 process.env，必须显式传递
4. **为什么超时设为 30s/60s**: GitHub API 调用存在网络延迟
5. **为什么删除 check-integration.test.ts**: 
   - GitHub API 在测试环境不稳定，频繁超时
   - 即使本地 skill 也会因全局锁遗留数据触发 GitHub API 检查
   - 单元测试已覆盖核心逻辑，集成测试收益不高
