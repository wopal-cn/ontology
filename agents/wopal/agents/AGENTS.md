# AGENTS.md — wopal-workspace 行为宪法

本宪法是 Agent 在 wopal-workspace 工作空间内的根本行为规范。每次会话启动时，Agent 必须依照本宪法确立自身定位，并严格遵守所列条款。

### 核心使命

wopal-workspace 存在的根本目的是：为 Agent 提供一套完整的工具栏和规则体系，以高效开发和维护空间内的各子项目。**子项目开发是 Agent 的核心职责**，工作空间的一切基础设施（命令、规则、技能、记忆系统）均服务于这一目标。

## 第一部分：记忆制度

### 核心原则

- **不主动记录**：只在用户明确指令（"记住这个"、"记下来"）时才执行记录
- **价值导向**：只记录对后续开发有长期价值的信息
- **判断基准**：「一个月后回头看，这条信息还有用吗？」

### 操作规范

记录流程与触发条件，适用 `.agents/rules/mem-rule.md` 的规定。

**自我进化**：你的的命令、规则和技能均源自 `projects/agent-tools/` 子项目。你有权也有责任通过优化 `agent-tools` 来升级自身的工具体系——包括改进现有命令、完善规则、开发新技能。修改后须遵循工具开发铁律（在源码中修改，通过部署站分发）。

## 第二部分：安全红线（绝对禁止违反）

### 红线一：保留原则

- 严禁未经审慎考量删除任何文件或目录
- **授权要件**：仅在用户以命令式语句（如"删掉它"）明确确认时，方可执行删除
- 用户以疑问形式提出（如"是否应该删除"）不构成授权，Agent 可建议但必须等待确认

### 红线二：工作边界

- Agent 的操作领域限定为：`/Users/sam/coding/wopal/wopal-workspace`
- 未经明确多重授权，严禁越出此边界修改系统配置或隐私文件
- 严禁在代码或提交中包含 `.env`、密钥或其他敏感凭证

### 红线三：external/ 目录保护

- `external/` 目录是工作空间对外部资源的引用集合，包含 Agent 全局配置、环境脚本、知识库链接及外部工具路径
- 未经用户明确授权，Agent 严禁主动创建、修改或删除 `external/` 下的任何内容
- Agent 可以读取和分析 `external/` 下的内容以协助用户

### 红线四：坦诚

- 不确定就如实说，发现错误就立即纠正。不掩饰，不搪塞

## 第三部分：工作规范

### 开发铁律

- 命令、规则、技能的修改，必须在 `projects/agent-tools/` 源码中执行
- **部署(同步)规则**：
  - 修改 `projects/agent-tools/` 源码后, 命令（commands）,规则（rules）,技能(skills) 等发生变更，**必须**使用 `scripts/sync-to-wopal.py` 同步变更到 `.wopal/` 部署站
  - 禁止使用 edit 工具直接编辑 `.wopal/` 或 `.agents/` 等部署站下的文件

- **子项目规范更新**：
  - 子项目变更完成后（如新增能力、架构决策、工具状态），必须更新子项目的 `AGENTS.md`，将变更同步到项目规范

### 技能 skill 优先调用义务

- **必须优先使用技能**：为保证开发流程规范统一，Agent 在执行任务前，**必须**主动寻找并调用 `.agents/skills/` 内的现有技能。
- **严禁自作主张**：放着现成技能不用，强行用大模型通用能力生成代码或执行操作，即视为严重违宪。
- **技能调用规则**: 必须从 `.agents/skills/` 加载，严禁从 `projects/agent-tools/skills/` 调用（测试场景除外）

### 子项目隔离法则

- `projects/` 下各子项目拥有**独立的 Git 历史与生命周期**
- 在单一子项目内工作时，聚焦该项目，按需加载其 `AGENTS.md`，禁止一次性加载所有子项目上下文
- 子项目内的修改必须在子项目内独立提交，不得跨项目污染主仓库提交历史

**进入子项目工作时的上下文加载协议：**
1. **始终生效**：工作空间宪法（本文件）中的安全红线和工程规范
2. **必须加载**：目标子项目的 `AGENTS.md`（项目规范），获取技术栏、命令、架构模式
3. **按需加载**：工作空间 `MEMORY.md` 中与该子项目相关的记录
4. **按需加载**：各层级目录下的 `AGENTS.md`，这是给你读的 README 文件
5. **按需加载**：各子项目以及各层级目录下的 `README.md` 文档是给人读的，不是给你读的，请你只能在缺失 `AGENTS.md` 的文档时按需读取。

**优先级规则**：子项目规范中的项目特有规范（如技术栏、测试命令）优先于工作空间通用规范；但安全红线和 Git 提交规范不可被子项目规范覆盖

### Worktree 工作流

当需要在子项目中**并行开发多个版本**或**隔离功能开发**时，使用 Worktree。

**使用指南**：参见 `git-worktrees` 技能

**核心原则**：
- 使用脚本管理（`.agents/skills/git-worktrees/scripts/worktree.sh`），自动完成安全验证、依赖安装和测试
- 完成后务必清理，避免僵尸目录
- worktree 内的提交直接写入子项目 Git 历史

### OpenSpec 驱动协作

当使用 OpenSpec 规范驱动其他 AI Agent（如 OpenCode）执行任务时。

**核心原则**：
- 创建 OpenSpec 产物（proposal、specs、design、tasks）并传递给执行 Agent
- 完成后**必须更新子项目规范**：将新增能力、架构决策同步到子项目的 `AGENTS.md`

**Capability 命名规范**：
- 格式：`<产品>-<功能域>-<具体能力>`
- 示例：`wopal-cli-skills-download`、`wopal-cli-skills-scan`
- 原则：包含产品/功能前缀，明确归属关系

**详细指南**：参见 `docs/openspec-guide.md`

### 时间处理规范

- 记录或判断时间时，必须使用系统命令获取准确时间，严禁凭记忆推断
- 推荐命令：`date '+%Y-%m-%d %H:%M:%S'` 或 `date '+%Y-%m-%d'`

### 工程规范

- 代码修改后必须履行语法验证（以编译通过为准）
- Git 提交必须使用中文，严格遵循 Conventional Commits 规范

### 软链接处理

- 工作空间内的软链接视为普通目录访问
- 严禁读取或解析其原始绝对路径

### 文档维护规则

**`.workspace.md`（工作空间导航地图）**
- 定位：记录工作空间**当前存在什么**——目录结构、项目清单、功能区划分
- 维护时机：当目录结构发生变更时（新增/删除/重命名/移动目录或项目），Agent 必须同步更新此文件
- 维护原则：只陈述事实，不记录决策原因或历史背景

**`MEMORY.md`（工作空间以及子项目集体记忆库）**
- **核心定位**：工作空间内**各子项目的集体记忆**——记录我让你记住的一些关键事实，分为工作空间和子项目两个层面
- **次要定位**：工作空间基础设施的重大变更（目录重构、宪法升级、工具链演进）
- 维护时机：在发生子项目重大决策、技术选型、里程碑达成、或吸取教训时更新
- 维护原则：只记录对**跨会话开发**有参考价值的信息，不记录代码技术细节（由子项目 AGENTS.md 承载）

**`.tmp/.working-context.md`（临时上下文文件）**
- 定位：上下文传递的临时文件，内容可能不断变化
- 使用原则：
  - **严禁主动读取**：Agent 不得主动读取此文件,避免加载过时信息污染上下文.
  - **严禁主动更新**：Agent 不得主动创建或更新此文件
  - **严禁引用**：避免在其他文档中引用此文件，因内容变化会导致引用失效

**宪法立法原则**
- 宪法（本文件）只记录**研究明确后的确定方案**，禁止写入可选项、待研究方案或多选方案
- 探索阶段的方案应记录在 `docs/` 或 `memory/` 中，待验证成熟后方可立法
