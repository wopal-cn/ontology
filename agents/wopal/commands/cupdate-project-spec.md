# 创建/更新项目规范

## 项目和目标: `$ARGUMENTS` -> [project] [target]

| 参数 | 必填 | 说明 |
|------|------|------|
| `project` | 否 | 项目路径，未提供则从上下文推断 |
| `target` | 否 | `agents` / `readme` / `both`，默认 `both` |

**示例**:
- `/cupdate-project-spec` → 推断项目，生成两个文件
- `/cupdate-project-spec wopal-cli` → 指定项目
- `/cupdate-project-spec wopal-cli agents` → 仅生成 AGENTS.md

---

## 执行流程

### 1. 确定目标项目

**有参数**: 使用参数

**无参数**: 从上下文推断
- 检查当前工作目录
- 检查最近操作的文件所属项目
- 检查会话中讨论的项目

```bash
pwd | grep -o 'projects/[^/]*'  # 提取项目名
```

### 2. 深度分析项目

**必须实际读取代码**，不是猜测：

```bash
# 目录结构
tree -L 3 -I 'node_modules|dist|.git'

# 技术栈
cat package.json 2>/dev/null || cat pyproject.toml 2>/dev/null || cat Cargo.toml 2>/dev/null

# 入口文件
cat src/index.ts src/main.ts src/cli.ts 2>/dev/null | head -100

# 核心逻辑（抽样 2-3 个）
find src -name "*.ts" -type f | head -5 | xargs cat
```

**识别**：
- 项目类型、技术栈、目录结构
- 关键入口点、核心逻辑位置
- 开发约束（从代码风格、注释、lint 配置提取）

### 3. 确认操作

```
将为 projects/<project> 生成/更新：
- AGENTS.md（面向 AI Agent）
- README.md（面向人类开发者）

确认？[Y/n]
```

### 4. 生成文档

## AGENTS.md 质量标准

### 核心原则

1. **架构优先** — 用图/表表达结构关系，不是目录清单
2. **定位清晰** — 说明项目在更大上下文中的角色
3. **精简实用** — 省略常规目录（如 `node_modules`、`dist`），只说非常规的
4. **约束明确** — 开发约束要具体，不要模糊描述

### 必要部分

| 部分 | 作用 | 要点 |
|------|------|------|
| **定位** | 一句话说明项目角色 | 在更大系统中的位置 |
| **架构** | 结构关系图/表 | 核心模块 + 数据流/依赖流 |
| **资源类型** | 如有多种资源类型 | 用表格归类 |
| **目录与归属** | 非常规目录说明 | 标注归属/用途 |
| **快速定位** | 搜索命令 | 常用定位方法 |
| **开发规范** | 部署/测试/约束 | 具体命令和规则 |

### 可选部分（按需）

- 项目特有模式（仅写此项目特有的）
- 关键模块详解（复杂项目才需要）

### 禁止

- ❌ 完整目录树罗列（常规目录省略）
- ❌ 模糊描述（"遵循最佳实践"）
- ❌ 过时的索引清单（无法维护的内容）

---

## README.md 质量标准

### 核心原则

1. **独立完整** — 视为独立项目介绍，不提及 monorepo 关系
2. **快速上手** — 安装 + 运行命令一目了然
3. **价值导向** — 说明能做什么，不是怎么实现

### 必要部分

| 部分 | 作用 |
|------|------|
| **简介** | 一句话核心价值 |
| **快速开始** | 安装 + 运行命令 |
| **核心功能** | 能做什么 |
| **技术栈** | 用到什么 |
| **License** | 许可证 |
