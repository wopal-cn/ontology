# 记忆管理规则

## 核心原则

只沉淀与空间优化、项目建设**高度相关**且具有**长期复用价值**的信息。

### 沉淀三问

1. 能避免重复研究吗？→ 记录发现 + 文件引用
2. 能避免再犯同样的错吗？→ 记录避坑方法（重方法，轻教训）
3. 能更好地理解用户吗？→ 记录用户偏好

## 不记录

- 通用知识（已知常识、可从文档/网络直接获取）
- 临时状态（版本号、当前配置值）
- 会话流水（做了什么、决策过程）
- 单纯的错误描述（无根因、无方法）

## 分类体系

长期记忆使用以下 7 个分类，无论写入何种存储都须遵守：

| 分类 | 标识符 | 默认权重 | 判断标准 |
|------|--------|----------|---------|
| 用户要求 | `requirement` | 0.95 | 用户明确要求必须遵守的规则/行为 |
| 用户档案 | `profile` | 0.90 | 用户个人身份和基本信息 |
| 避坑方法 | `gotcha` | 0.85 | 踩过的坑、不要这样做（反面教训） |
| 经验 | `experience` | 0.85 | 验证有效的最佳实践（正面总结） |
| 偏好 | `preference` | 0.80 | 用户倾向但非强制 |
| 事实 | `fact` | 0.75 | 空间/项目特有的客观事实（通用可查的不记录） |
| 知识 | `knowledge` | 0.70 | 空间/项目特有的技术理解（通用可查的不记录） |

区分要点：`preference` 是"倾向"，`requirement` 是"必须"；`gotcha` 是反面教训，`experience` 是正面实践。

## 写入格式（结论前置六原则）

所有记忆写入操作必须遵守：

1. **结论前置**：第一行必须是结论/规则/核心内容，背景和细节放后面
2. **直接描述**：不要 `## [类别]:` 前缀（类别由字段或标题标识）
3. **格式简洁**：用普通文本，不要 `**粗体标签**：` 格式
4. **背景必要**：包含"为什么这很重要"或"在什么场景下有用"
5. **内容完整**：保留必要的代码/路径/命令
6. **不截断**：保持完整，去除冗余描述

## 工具参数规范

`memory_manage` 工具添加记忆时必须填写以下字段：

| 字段 | 参数名 | 说明 |
|------|--------|------|
| 分类 | `category` | 7 个合法分类之一（见上方分类体系） |
| 正文 | `text` | 遵循结论前置六原则 |
| 重要性 | `importance` | 根据分类默认权重调整 |
| 项目 | `project` | 记忆所属项目，搜索时按项目过滤。空间级通用记忆用 `wopal-space`；项目特有记忆用具体项目名（如 `space-flow`、`ontology`、`wopal-cli`） |
| 标签 | `tags` | 逗号分隔的关键词，用于精确检索 |

**注意**：标签字段名是 `tags`（不是 `concepts`），存于记忆的 `tags` 字段。缺少项目或标签会降低搜索精确性。

## 沉淀位置

| 信息性质 | 主存储 | 降级存储 | 方式 |
|----------|--------|----------|------|
| 长期知识/经验/避坑 | 长期记忆（`memory_manage`） | `MEMORY.md` | 工具写入 / 手动追加 |
| 用户偏好 | `USER.md` | — | 直接编辑（需授权） |
| 工作规则/约定 | `AGENTS.md` | — | 直接编辑（需授权） |
| 项目特定知识 | 项目 `AGENTS.md` | — | 直接编辑（需授权） |
| 核心行为特质 | 灵魂（系统提示词） | — | 修改 ontology 后部署（需授权） |
| 短期会话记忆 | `memory/diary/` | — | 用户手动触发或者系统定时触发 |

> **降级机制**：可用 `memory_manage` 工具时，通过该工具写入长期记忆数据库；不可用时，按同样的分类标准和格式写入 `MEMORY.md`。两种方式的沉淀标准完全一致，区别仅在存储载体。

## 沉淀时机

- 用户说"记下来/记录知识" → 从分析成果中提取关键事实写入长期记忆
- 发现重复出现的信息 → 主动提醒沉淀价值，等用户确认后执行
- 用户偏好含"应该/必须" → 提炼到宪法或长期记忆 `requirement`

## 蒸馏筛选规范

蒸馏生成候选后，必须**全量展示**给用户，并逐条附筛选建议。用户最终拍板，严禁跳过展示。

### 全量展示义务

蒸馏产出的每条候选记忆都必须完整展示，不得以任何理由跳过。用户需要通过展示内容监督筛选是否合理。

### 逐条筛选判断（6 条义务）

每条候选必须依次通过以下判断，未通过的在展示时标注原因+建议：

| # | 判断 | 标准 | 处理 |
|---|------|------|------|
| (1) | 重复检查 | 与已有记忆内容重复或高度相似 | 建议跳过或合并，注明重复对象 |
| (2) | 通用性过滤 | 通用可查知识（框架文档、语言特性、公共 API） | 建议跳过，注明"可从文档获取" |
| (3) | 时效性判断 | 已修复/已解决的问题、已完成的临时工作 | 建议跳过，注明"已修复/已完成，未来 Agent 读到无指导意义" |
| (4) | 会话流水过滤 | 验证流程、调试过程、决策过程、操作记录 | 建议跳过，注明"会话流水，无长期复用价值" |
| (5) | 价值必要性检验 | 未来 Agent 读到后能否避免错误或做出更好决策？ | 不能则建议跳过，注明具体原因 |
| (6) | 格式合规 | 符合结论前置六原则 | 不合规则修正后再建议保留 |

### 展示格式

每条候选附筛选建议，格式：

```
[0] <候选内容>
   → 筛选建议：保留/跳过/合并 — <理由>
```

### 用户决策

- 展示完毕后等待用户选择（全写/精选/取消）
- 严禁跳过展示直接 confirm
