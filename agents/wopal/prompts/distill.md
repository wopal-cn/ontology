# 记忆提取 Prompt

**必须输出 JSON 格式。禁止输出 `<system-reminder>` 或其他格式。**

输出模板：
{"title": "简短标题（不超过20字）", "memories": [{"category": "knowledge", "body": "标题\n\n核心内容...", "concepts": ["tag"]}]}

如果无记忆可提取，输出：{"title": "无内容", "memories": []}

# 提取标准

## 核心原则

你的目标是提取**对未来会话有持续指导价值**的记忆，不是记录本次对话发生了什么。宁可漏掉也不要记流水账。

**优先提取**：
- 从用户不满、批评、纠正中提炼的深层偏好和做事原则
- 跨场景可复用的经验、方法论、判断标准
- 反复出现的模式（用户多次强调同一件事 → 高信号）
- 踩坑后的教训（错误 + 原因 + 正确做法，三者缺一不可）

## 什么是值得记住的？

- **长期有效**：未来会话中仍有用的信息，不随时间过时
- **跨场景可复用**：不只是解决当前问题，而是形成可迁移的做法
- **个性化信息**：特定于该用户的信息，非通用领域知识
- **具体明确**：有具体细节，非模糊概括
- **可独立理解**：第三方无需原对话上下文也能理解

## 什么不值得记住？

**坚决不记**：
- **已解决的一次性问题**：本次会话中提出、讨论、解决的技术问题——代码本身就是最权威的文档，不需要记忆复述
- **代码变更记录**："修改了 X 文件"、"重构了 Y 函数"——这是 git log 的事
- **当前会话已消化的信息**：对话中已经达成共识并落地的决策，未来不会再需要
- **通用知识**：任何人都知道的信息，或能通过文档/搜索轻易查到的
- **瞬时状态**："Issue #N 处于 X 阶段"、"当前状态是 Y"——状态很快过时
- **短期待办**："下一步验证 X"——这是任务管理，不是知识
- **进度流水**："已完成 X，正在做 Y"——除非包含可复用的经验或决策
- **工具输出**：错误日志、样板代码、API 响应
- **回忆查询**："你还记得 X 吗？"——这是检索请求，不是新信息
- **退化引用**：用户模糊提到某事时，不要编造细节
- **调试临时态**：临时修复、调试日志配置、临时环境变量
- **已固化的设计决策**：已落实为代码、配置文件、提示词文件或系统提示词的要求和方案——文件本身就是记录

# 分类体系（7 类）

| 中文标签 | 英文 category | 定义 | 判断问题 |
|---------|--------------|------|---------|
| 用户画像 | profile | 用户身份、静态属性 | "用户是谁？" |
| 用户偏好 | preference | 用户习惯、倾向、风格 | "用户偏好什么？" |
| 技术知识 | knowledge | 调研结果、技术事实、参考资料 | "这是什么/怎么工作？" |
| 项目事实 | fact | 调研结论、客观发现、项目决策结果 | "发现了什么/决定了什么？" |
| 避坑方法 | gotcha | 历史错误、陷阱、预防措施（必须有踩坑经历） | "怎么避免踩坑？" |
| 实践经验 | experience | 可复用流程、工作模式、方法论 | "什么流程可以复用？" |
| 用户要求 | requirement | 用户反复强调的约束、明确规则 | "用户要求必须怎么做？" |

# 分类判断树

判断一条信息属于哪个类别，**按优先级从上到下**检查：

1. **用户要求？**
   用户明确说"必须/不要/以后都..." → requirement
   例："必须用绝对路径"、"禁止自动 push"
   ⚠️ 注意：只有"规则"没有"踩坑经历" → 用户要求（不是 gotcha）

2. **历史踩坑？**
   用户提到过去遇到的问题 + 解决方案 → gotcha
   例："之前踩过坑，用 {...} 展开就好了"
   ⚠️ 关键：必须包含"过去遇到过"的描述

3. **可复用流程/模式？**
   能跨场景复用的操作步骤、工作策略 → experience
   例："fc-local 优先，firecrawl 备选"
   例："apply_patch 失败就改用 write"

4. **技术事实/工作原理？**
   技术原理、API 枚举值、配置说明、参考资料路径 → knowledge
   例："Part type 是 tool 不是 tool_call"
   例："LanceDB FTS 默认英文分词"
   ⚠️ 注意："调研发现 X 工作方式" → knowledge；"调研结论 X 方案最优" → fact

5. **调研结论/项目决策？**
   用户得出"X 方案最优" → fact
   项目选型决策、架构决策 → fact
   例："决定用 docs/ 集中管理文档"

6. **用户偏好/习惯？**
   风格、倾向、习惯（不是强制规则） → preference
   例："不加 emoji"、"不用 class 组件"

7. **用户身份/静态属性？**
   职业、技术栈、背景 → profile

# 常见混淆

| 错误分类 | 正确分类 | 原因 |
|---------|---------|------|
| "用户偏好 X" | 用户偏好 | 不是画像 |
| "调研发现 X 工作方式" | 技术知识 | 不是事实 |
| "调研结论 X 方案最优" | 项目事实 | 不是知识 |
| "遇到问题 A，用方案 B" | 避坑方法 | 必须有历史踩坑经历 |
| "处理某类问题的一般流程" | 实践经验 | 不是避坑方法 |
| "用户明确说不要 X" | 用户要求 | 不是偏好 |
| "技术规则约束（无踩坑经历）" | 用户要求 | gotcha 必须有踩坑经历 |

# Body 格式规范

每条记忆必须是一个 **self-contained 的结构化文本**，能被第三方独立理解。

## 核心原则

1. **直接描述**：第一行直接写核心内容，不要 `## [类别]:` 前缀（类别由 category 字段标识）
2. **格式简洁**：用普通文本，不要 `**粗体标签**：` 格式，需要标注时用 `标签：` 即可
3. **背景必要**：必须包含"为什么这很重要"或"在什么场景下有用"
4. **内容完整**：保留必要的代码/路径/命令，不要过度精简
5. **不截断**：保持完整，但去除冗余描述

## 格式模板

### 用户画像
{"category": "profile", "body": "<身份描述>\n\n- 职业: ...\n- 技术栈: ...", "concepts": ["背景"]}

### 用户偏好
{"category": "preference", "body": "<偏好描述>\n\n背景：...\n适用：...", "concepts": ["偏好"]}

### 技术知识
{"category": "knowledge", "body": "<精确主题>\n\n<核心内容>\n\n来源：...", "concepts": ["reference"]}

### 项目事实
{"category": "fact", "body": "<发现/决策>\n\n<结论和细节>", "concepts": ["how-it-works"]}

### 避坑方法
{"category": "gotcha", "body": "<问题描述>\n\n问题：...\n方案：...\n适用：...", "concepts": ["gotcha", "problem-solution"]}

### 实践经验
{"category": "experience", "body": "<流程/模式描述>\n\n流程：...\n原因：...", "concepts": ["pattern"]}

### 用户要求
{"category": "requirement", "body": "<规则描述>\n\n背景：...\n适用：...", "concepts": ["user-rule"]}

# Few-shot 示例

用户说："我之前用 tail -f 监控日志，界面卡死了，以后都用 tail -n 30"

输出：
{"memories": [{"category": "gotcha", "body": "tail -f 导致界面卡死\n\n问题：tail -f 监控日志会导致 OpenCode 界面挂起\n方案：使用 tail -n 30 代替，或定期读取文件内容\n适用：OpenCode 插件开发调试时监控日志", "concepts": ["gotcha"]}]}

用户说："以后代码提交前先让我评审"

输出：
{"memories": [{"category": "requirement", "body": "代码提交前必须先评审\n\n背景：用户希望对代码变更有审核权\n要求：Git commit 前必须先提供变更列表供用户评审，只有用户明确要求时才执行 commit/push\n适用：所有 Git 提交场景", "concepts": ["user-rule"]}]}

# 质量检查

提取后对每条记忆自检：

1. **值得记吗？** — 未来会话还会需要这条信息吗？如果答案不确定，丢弃
2. **标题精确吗？** — 标题是否准确描述"这是什么"
3. **有背景吗？** — 是否包含"为什么这很重要"
4. **第三方能理解吗？** — 没看过原对话的人能理解吗
5. **信息完整吗？** — 是否缺少关键代码/路径/命令
6. **是流水账吗？** — 如果去掉这条记忆，未来会话能通过代码/文档/搜索自行获取，则丢弃

**不合格则丢弃或重试**。

# 注意事项

- 输出语言应与对话中主要语言匹配
- 宁可少记也不多记。3 条高质量记忆 > 8 条流水账
- 每条记忆必须能独立理解，不能假设读者看过原对话
- 用户在对话中的**不满、纠正、反复强调**是高价值信号——提炼背后的原则，而不是记录表面事件
